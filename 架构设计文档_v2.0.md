## 1. 项目概述与设计目标

### 1.1 场景定义

构建一个部署在边缘端的模型推理服务，具备以下特征：
- **异构硬件**：支持 NVIDIA RTX 4090、纯 CPU（x86/ARM）、瑞芯微（Rockchip）NPU
- **多并发负载**：需支持多个用户或客户端同时请求
- **高稳定性**：边缘环境无人值守，要求 7×24 小时稳定运行

### 1.2 核心目标

| 维度 | 目标 |
|------|------|
| 性能 | 充分利用硬件算力（GPU/NPU），单请求延迟 < 10s |
| 通用性 | 一套代码底座适配多种硬件后端 |
| 安全性 | 杜绝内存泄漏与并发数据竞争，零 FFI panic |
| 可观测性 | 结构化日志 + Prometheus 指标，完整链路追踪 |
| 自愈能力 | 看门狗 + 自动恢复，无需人工干预 |
| 资源治理 | 显存/内存全程监控，阈值告警 |

#### 3.2.2 Request Scheduler (Slot Pool + Continuous Batching)

**职责**：
- 维护固定大小的 Slot Pool（可配置，默认 16 slots）
- 每个 Slot 独立管理一个推理会话的生命周期
- 实现 Continuous Batching 批处理调度器

**Slot 状态机**：

```
                    ┌─────────────┐
         ┌─────────►│   IDLE      │◄────────────┐
         │          └──────┬──────┘             │
         │                 │  acquire()         │ release()
         │                 ▼                    │
         │          ┌─────────────┐             │
         │          │  PENDING    │             │
         │          │ (queued)    │             │
         │          └──────┬──────┘             │
         │                 │  batch()           │
         │                 ▼                    │
         │          ┌─────────────┐             │
         │          │ INFERENCING │             │
         │          │ (streaming) │             │
         │          └──────┬──────┘             │
         │                 │  complete/timeout  │
         │                 ▼                    │
         │          ┌─────────────┐             │
         └──────────│  RELEASING  │─────────────┘
                    └─────────────┘
```

**Continuous Batching 调度策略**：

```rust
// 调度器伪代码
loop {
    // 1. 收集 PENDING 状态的 slots
    let pending: Vec<SlotId> = slot_pool.pending_slots();

    // 2. 合并相同批次的 slots（共享前缀）
    let batch = merge_to_batch(pending, max_batch_size = 8);

    // 3. 执行批处理推理
    let results = inference_backend.forward_batch(batch).await;

    // 4. 分发结果
    for (slot_id, result) in results {
        slot_pool.complete(slot_id, result);
    }

    // 5. 动态调整批次大小
    adjust_batch_size_based_on_throughput();
}
```

**超时控制**（三级超时）：
- `prompt_processing_timeout`: 5s（Tokenize + Prefill）
- `first_token_timeout`: 30s（首 token 生成）
- `per_token_timeout`: 2s（每个后续 token）
- `global_timeout`: 120s（整个请求）

**超时处理**：
```rust
match tokio::time::timeout(global_timeout, inference).await {
    Ok(Ok(result)) => send_result(result),
    Ok(Err(e)) => send_error(ErrorCode::InferenceError, e),
    Err(_) => {
        // 超时：标记 slot 为 TIMEOUT，强制 cancel
        slot_pool.timeout_cancel(slot_id);
        send_error(ErrorCode::Timeout, "Inference exceeded global timeout")
    }
}
```

#### 3.2.3 Inference Abstraction Layer (FFI Boundary)

**核心设计原则**：
- 所有 llama.cpp C API 调用封装在 `unsafe` 块内
- 通过 Rust wrapper 结构体管理指针生命周期
- 使用 `ManuallyDrop` 防止意外析构

**关键类型定义**：

```rust
// Model 句柄封装
pub struct LlamaModel {
    ctx: *mut llama_context,
    _pinned: PhantomData<&'static mut LlamaModel>,
}

unsafe impl Send for LlamaModel {}
unsafe impl Sync for LlamaModel {}

impl LlamaModel {
    pub fn from_file(model_path: &Path) -> Result<Self, InferenceError> {
        let c_path = CString::new(model_path.as_os_str().as_encoded_bytes())
            .map_err(|_| InferenceError::InvalidPath)?;

        let params = llama_model_default_params();
        let ctx = unsafe { llama_load_model_file(c_path.as_ptr(), params) };

        if ctx.is_null() {
            return Err(InferenceError::ModelLoadFailed(model_path.display().to_string()));
        }

        Ok(Self { ctx, _pinned: PhantomData })
    }
}

impl Drop for LlamaModel {
    fn drop(&mut self) {
        unsafe { llama_free(self.ctx) }
    }
}

// Backend trait：统一抽象
pub trait InferenceBackend: Send + Sync {
    fn name(&self) -> &str;  // "cuda", "cpu", "rkllm"
    fn forward(&self, batch: &InferenceBatch) -> Result<Vec<Token>, InferenceError>;
    fn health_check(&self) -> bool;
    fn memory_stats(&self) -> MemoryStats;
}
```

**FFI 调用超时包装**：

```rust
pub struct InferenceEngine {
    backend: Arc<dyn InferenceBackend>,
    watchdog_tx: Arc<WatchdogChannel>,
}

impl InferenceEngine {
    pub async fn predict(&self, prompt: &str, config: &InferenceConfig)
        -> Result<InferenceResult, InferenceError>
    {
        let timeout = config.timeout.unwrap_or(DEFAULT_TIMEOUT);

        tokio::time::timeout(timeout, self._predict_internal(prompt, config))
            .await
            .map_err(|_| InferenceError::Timeout)?
    }

    async fn _predict_internal(&self, prompt: &str, config: &InferenceConfig)
        -> Result<InferenceResult, InferenceError>
    {
        // 1. Tokenize
        let tokens = self.tokenize(prompt)?;

        // 2. Validate prompt length
        if tokens.len() > MAX_PROMPT_TOKENS {
            return Err(InferenceError::PromptTooLong(tokens.len(), MAX_PROMPT_TOKENS));
        }

        // 3. Build batch
        let batch = InferenceBatch::new(tokens, config);

        // 4. Forward (wrapped in watchdog)
        let watchdog = self.watchdog_tx.start(config.timeout.unwrap_or(DEFAULT_TIMEOUT));

        let result = self.backend.forward(&batch).await;

        watchdog.stop();

        result
    }
}
```

#### 3.2.4 Backend Factory

**职责**：根据配置或自动检测，选择合适的推理后端。

```rust
pub enum BackendType {
    Cuda,
    Cpu,
    Rkllm,
}

pub struct BackendFactory;

impl BackendFactory {
    pub fn create(backend_type: BackendType, model_path: &Path)
        -> Result<Arc<dyn InferenceBackend>, InferenceError>
    {
        match backend_type {
            BackendType::Cuda => {
                #[cfg(feature = "cuda")]
                return Ok(Arc::new(CudaBackend::new(model_path)?));

                #[cfg(not(feature = "cuda"))]
                return Err(InferenceError::BackendNotAvailable("CUDA"));
            }
            BackendType::Cpu => Ok(Arc::new(CpuBackend::new(model_path)?)),
            BackendType::Rkllm => {
                #[cfg(feature = "rkllm")]
                return Ok(Arc::new(RkllmBackend::new(model_path)?));

                #[cfg(not(feature = "rkllm"))]
                return Err(InferenceError::BackendNotAvailable("RKLLM"));
            }
        }
    }

    pub fn auto_detect() -> BackendType {
        // 检测优先级：CUDA > RKLLM > CPU
        #[cfg(feature = "cuda")]
        if Self::cuda_available() {
            return BackendType::Cuda;
        }

        #[cfg(feature = "rkllm")]
        if Self::rkllm_available() {
            return BackendType::Rkllm;
        }

        BackendType::Cpu
    }
}
```

## 5. 安全设计

### 5.1 认证与授权

```rust
// API Key 认证中间件
async fn auth_middleware(
    headers: HeaderMap,
    config: &AuthConfig,
) -> Result<ClientId, AuthError> {
    let token = headers
        .get("Authorization")
        .and_then(|v| v.to_str().ok())
        .and_then(|v| v.strip_prefix("Bearer "))
        .ok_or(AuthError::MissingToken)?;

    let client_id = config
        .validate_api_key(token)
        .ok_or(AuthError::InvalidToken)?;

    Ok(client_id)
}

// 速率限制（token bucket per client）
async fn rate_limit_middleware(
    client_id: ClientId,
    rate_limiter: &RateLimiter,
) -> Result<(), RateLimitError> {
    let (allowed, remaining, reset_at) = rate_limiter.check(client_id).await;

    if !allowed {
        return Err(RateLimitError::Exceeded {
            retry_after: reset_at - now(),
        });
    }

    Ok(())
}
```

### 5.2 Prompt 安全

- **长度限制**：最大 8192 tokens，超限返回 400
- **上下文比例**：Prompt 不能超过模型上下文长度的 75%
- **注入检测**：检测常见 prompt 注入模式

### 5.3 模型保护

- 模型文件存储在加密分区（dm-crypt）
- 运行时设置 `prctl(PR_SET_DUMPABLE, 0)` 禁止 core dump
- 模型加载后立即关闭文件句柄

## 6. 边缘场景特殊设计

### 6.1 自愈机制

```rust
// 看门狗：监控推理是否卡死
struct Watchdog {
    last_activity: AtomicU64,
    timeout: Duration,
    panic_sender: Arc<watchdog::PanicChannel>,
}

impl Watchdog {
    fn start(&self) -> WatchdogGuard {
        let self_ptr = self as *const Self;
        let timeout = self.timeout;

        // 启动后台监控任务
        tokio::spawn(async move {
            let interval = Duration::from_secs(30);
            loop {
                tokio::time::sleep(interval).await;

                let elapsed = Instant::now() - last_activity;
                if elapsed > timeout {
                    // 触发看门狗重启
                    panic_sender.send(WatchdogTimeout {
                        elapsed,
                        last_activity,
                    });
                }
            }
        });

        WatchdogGuard { parent: self_ptr }
    }
}

// 渐进式重启流程
async fn graceful_restart(engine: Arc<InferenceEngine>) {
    // 1. 停止接受新请求
    set_health_status("maintenance").await;

    // 2. 等待正在进行的请求完成（最多 60s）
    let pending = slot_pool.pending_count();
    if pending > 0 {
        tokio::time::sleep(Duration::from_secs(60)).await;
    }

    // 3. 强制取消剩余请求
    slot_pool.cancel_all();

    // 4. 重新加载模型
    engine.reload().await;

    // 5. 恢复服务
    set_health_status("healthy").await;
}
```

### 6.2 显存健康管理

```rust
// 显存监控：每分钟记录一次
async fn memory_monitor_task(engine: Arc<InferenceEngine>) {
    let mut last_used = 0u64;

    loop {
        tokio::time::sleep(Duration::from_secs(60)).await;

        let stats = engine.memory_stats();
        metrics::gauge!("gpu_memory_used_bytes", stats.used as f64);

        // 检测显存泄漏
        if stats.used > last_used + 50 * 1024 * 1024 {  // 50MB 增长
            tracing::warn!(
                memory_growth_bytes = stats.used - last_used,
                "Potential memory leak detected, triggering model reload"
            );

            // 触发渐进式重启
            graceful_restart(engine.clone()).await;
        }

        last_used = stats.used;
    }
}
```

### 6.3 断电恢复（幂等设计）

```rust
// 请求去重：基于 X-Request-ID
async fn dedup_middleware(
    request_id: String,
    cache: Arc<DeduplicationCache>,
) -> Result<Option<Response>, Error> {
    if let Some(cached) = cache.get(&request_id).await {
        tracing::info!(request_id = %request_id, "Returning cached response");
        return Ok(Some(cached));
    }
    Ok(None)
}

// 请求完成后的缓存处理
async fn cache_response(request_id: String, response: Response, cache: Arc<DeduplicationCache>) {
    cache.set(&request_id, response, ttl = Duration::from_secs(3600)).await;
}
```

## 11. 附录

### 11.1 llama.cpp Tensor Parallelism 支持状态

**当前状态**：llama.cpp 原生对多卡张量并行的支持**尚未完全成熟**。

| 方案 | 状态 | 说明 |
|------|------|------|
| llama.cpp CUDA 多卡 | 实验性 | 社区有分支支持，但非官方主线 |
| NCCL 集成 | 未支持 | 需自行集成或使用第三方分支 |
| 单卡 int4 量化 | **已支持** | Q4_K_M 等格式可在单卡 24GB 运行 |

**决策**：优先确保单卡 int4 量化版本稳定运行，多卡 TP 列入后续计划。

### 11.2 单卡 int4 量化运行目标

| 模型示例 | 量化格式 | 显存需求 | 单4090可行性 |
|----------|----------|----------|--------------|
| Qwen3-35B-A3B | Q4_K_M | ~20GB | ✅ 可运行 |
| LLaMA3-8B | Q4_K_M | ~5GB | ✅ 宽松 |
| LLaMA3-70B | Q4_K_M | ~40GB | ❌ 需多卡 |

**验证目标**：Phase 0/1 验证期内，确认单卡可加载并推理 int4 量化模型。

### 11.3 多卡 Tensor Parallelism 后续计划

| 阶段 | 内容 | 前置条件 |
|------|------|----------|
| Phase 4+ | 调研 llama.cpp 多卡分支（如社区 TP 分支） | Phase 3 完成 |
| Phase 4+ | 集成 NCCL 或等效集合通信库 | CUDA 环境验证 |
| Phase 4+ | 实现 CudaMultiBackend + Partition 管理 | 单卡后端稳定 |
| Phase 4+ | 双卡负载均衡 + 故障切换 | 多卡验证完成 |

**风险提示**：若 llama.cpp 官方长期不提供 TP 支持，需评估：
1. 使用第三方分支的维护成本
2. 切换到 vLLM（或等待 vLLM Rust 版本）的可行性
3. 模型并行（Pipeline Parallelism）作为替代方案
